MiddleOut - Architecture Approaches

Three approaches to structuring the app, with trade-offs for each

A

Recommended: Pure AppKit + SwiftUI Settings

Architecture: AppDelegate-driven lifecycle, NSWindow for progress, SwiftUI only for Settings view

Pros

  • Full control over window lifecycle (hide/show/close without quit)
  • NSPanel for progress popup — native, lightweight, auto-dismiss
  • Proven pattern for "agent" apps (no Dock, no Menu Bar)
  • Easiest to handle "relaunch = show settings" behavior

Cons

  • More AppKit boilerplate vs pure SwiftUI
  • Two UI paradigms (AppKit windows + SwiftUI views)
B

Pure SwiftUI App Lifecycle

Architecture: @main App struct with WindowGroup, SwiftUI-managed windows

Pros

  • Modern, less code
  • Consistent UI framework throughout

Cons

  • Poor control over "agent" lifecycle — hard to suppress initial window
  • WindowGroup doesn't support "close window but keep app alive" cleanly
  • Progress popup harder to implement as floating panel
  • Relaunch detection requires workarounds
C

Hybrid: SwiftUI App + AppDelegate Adapter

Architecture: SwiftUI @main with @NSApplicationDelegateAdaptor, mix of both

Pros

  • SwiftUI entry point with AppKit escape hatch
  • Can use Settings scene for preferences (macOS 14+)

Cons

  • Two lifecycle owners — subtle conflicts possible
  • Settings scene not available on macOS 13
  • Still needs AppKit for progress panel and agent behavior

My Recommendation: Option A

For a "pure background agent" app like MiddleOut, AppDelegate-driven lifecycle is the gold standard on macOS. It gives us precise control over: no Dock icon, no Menu Bar icon, window show/hide on relaunch, floating progress panel, and clean quit. SwiftUI is excellent for the settings UI, but shouldn't own the app lifecycle.
